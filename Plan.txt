Methys Project Plan
===================

Purpose
-------
Methys provides a Python-facing orchestration API for a C-powered computation
engine, with an HTML-first frontend definition layer. It is not a Streamlit
application and does not provide a web server itself.

Repository structure
--------------------
src/methys/
  __init__.py        Public Python API
  frontend.py        HTML, CSS, and JavaScript block composition
  native.py          Future native-binding boundary

native/
  include/methys/    Stable public C headers
  src/               C implementations
  CMakeLists.txt     Native shared-library build

tests/               Python API tests
docs/                Architecture and build notes

Implementation principles
-------------------------
1. Keep algorithms, numerical routines, state, and memory management in C.
2. Keep Python as a small, ergonomic API and binding layer.
3. Let callers define frontend blocks with raw HTML and optional CSS/JavaScript.
4. Let the embedding application choose web serving, routing, and event transport.
5. Execute PostgreSQL and MongoDB queries through native drivers with bound
   parameters, never Python driver dependencies.
6. Treat user-provided browser code as unsafe unless the embedding application
   sanitises or otherwise trusts it.

Initial milestones
------------------
1. Build the native core and expose its version through a Python binding.
2. Define error handling, owned-memory rules, and opaque C handles.
3. Add a C query executor using libpq and the MongoDB C driver.
4. Add the first native data structure and calculation API.
5. Add frontend event contracts between browser code and the host application.
6. Package platform-specific native artifacts with Python releases.
